1. System Architecture

The system consists of two web applications served by a single Python server, deployed to Railway and accessible via custom subdomains on shakopeelions.com. Member data is sourced from Neon CRM via REST API and cached server-side.

Deployments are triggered automatically when code is pushed to the main branch of the GitHub repository. Railway detects Python via requirements.txt and Procfile.

⚠ Railway uses an ephemeral filesystem — the member disk cache (members_cache.json) persists within a deployment but is wiped on new deploys. The first browser load after a new deployment will trigger a full Neon reload (~2-3 minutes).

Member Load Process

  1. Fetch all account IDs from GET /v2/accounts?userType=INDIVIDUAL (paginated, 200/page)
  2. Fetch each account individually in batches of 10 concurrent requests
  3. Filter to accounts with non-empty accountCurrentMembershipStatus
  4. Deduplicate by firstName+lastName+email, keeping active over lapsed
  5. Fetch memberships for each member to get type and expiry date
  6. Sort alphabetically by last name, save to in-memory cache and disk

Total time: approximately 2–3 minutes for ~120 members. Cache TTL is 12 hours.

8. Authentication

Dashboard (HTTP Basic Auth)

The dashboard and all /v2/ API proxy endpoints require HTTP Basic Auth. The browser prompts for credentials on first visit and caches them for the session. Credentials are configured via APP_USER_* environment variables in Railway.

Check-in App (No Auth)

The check-in app at /attendance/ requires no authentication — it's designed to run on a shared iPad at meetings. The member data it receives is intentionally limited (no email, phone, address, or sensitive flags).

Check-in Admin PIN

The Admin section of the check-in app is protected by a 4-digit PIN. This controls access to board member list upload, attendance records, and data reset.

⚠ Admin PIN is sensitive — do not include in user-facing documentation.

9. Application Components

serve.py — Python HTTP Server

Single-file Python server. No external dependencies beyond the standard library. Handles: static file serving, Neon API proxying, member caching, Basic Auth, and Host-header-based routing.

lions-dashboard.html — Member Dashboard

Single HTML file. All JS inline. Connects to /members for data and /v2/* for editing. Features: member table with filtering/sorting/pagination, member detail modal, custom flag editing (hardship, cash/check, exit status), CSV export, billing statement generation (Print Bill / Print All Bills).

attendance/ — Check-in App (React/Vite)

React app built with Vite. Source in attendance/src/App.jsx. Built output in attendance/dist/ (committed to repo so Railway can serve it without a build step). Features: board and general meeting check-in with on-screen keyboard, admin panel, member drawing (random 10%).

To rebuild after changes: cd attendance && npm run build, then commit the dist/ folder.

attendance/LionsAttendance.jsx — Original Source

The original standalone attendance app built by John (Lions member). Kept in lionsclub/attendance/ as reference. The integrated version lives in dashboard/attendance/src/App.jsx.

Arrears Calculation

For expired members, arrears are calculated as one cycle per 6-month block elapsed since expiry, plus one current cycle to reinstate. Each cycle is billed at the member's rate.

Billing Statements

Printable PDF billing statements can be generated from the dashboard:

  • Print Bills button — generates statements for all members where status = Expired AND Cash/Check flag = true
  • Print Bill in member modal — generates a statement for a single member

Statements use a #8 double-window envelope layout. Return address (top window) and recipient address (bottom window) are positioned for standard fold-and-insert mailing.
